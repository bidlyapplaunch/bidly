## P0 — CRITICAL (correctness, data integrity, security regressions)

### Winner processing & auction end (the highest-risk cluster)
- **BACKEND-01 / SVC-03 / SVC-05** — There are **two** end-of-auction systems running simultaneously: the 5s poller in `server.js:1064-1164` (picks winner as `bidHistory[last]`, sends `sendAuctionWonNotification`) **and** `winnerProcessingService` (picks winner by sorted amount, sends `sendWinnerNotification`). They can pick different winners and **both** send email. Worse, `auctionEndService.js` is a **third, broken** path wired into `processEndedAuctions` that calls a non-existent `getShopifyService().getClient()` and throws on every call.
  **Fix:** One service owns auction-end. Atomically claim the auction (`findOneAndUpdate({_id, winnerProcessed:false}, {$set:{winnerProcessingLock:true}})`) before any side effect. Delete `auctionEndService.js` and the duplicate email block in `server.js`. Use `determineWinner` (sorted) as the single source of truth.
- **BACKEND-08** — `completedAt` is set by `auctionEndService`/queried by `processEndedAuctions`, but **not declared in `models/Auction.js`** (there's even an index on the phantom field). `strict:true` drops it on save → ended auctions never marked processed → **infinite reprocessing**.
  **Fix:** Add `completedAt: { type: Date, default: null }` to the schema; verify it persists.
- **BACKEND-07** — `buyNow` (`auctionController.js:1170-1198`) does 3 non-atomic saves (`active` → addBid → `ended`). Race-prone; concurrent Buy-Now/bid can double-win.
  **Fix:** Single guarded `findOneAndUpdate` like `placeBid` already uses.
- **BACKEND-14** — `placeBid` re-`save()`s the doc after its atomic update to apply popcorn extension (`auctionController.js:913-924`), reintroducing a race.
  **Fix:** Apply the extension via `findOneAndUpdate` with `$max` on `endTime`, not a full save.

### Live security holes (NEW — not in AUDIT.md)
- **SVC-01** — `POST /api/billing/override {plan:"enterprise"}` is mounted with only `identifyStore` (no auth). **Free Enterprise for anyone.** Same primitive in `debugRoutes.js:206` `/set-plan`. → require `requireAuth` on all billing mutations.
- **SVC-02** — `debugRoutes` mounted unconditionally in prod (`server.js:701`). Exposes token previews/scopes per store, `DELETE /clear-store/:shop` (force-uninstall any store), `/set-plan`. → gate behind `NODE_ENV !== 'production'` + auth.
- **SVC-08** — `POST /api/customers/:id/bid` has **no** `verifyCustomerToken` (every sibling route does). Anyone can forge bid history / inflate `auctionsWon`. → add token check + owner match.
- **SVC-09** — App Proxy storefront bid/buy-now endpoints (`appProxyRoutes.js`) **don't validate the Shopify proxy HMAC** and trust `logged_in_customer_id` from the query string → bid as any customer. → verify proxy signature, derive identity from verified session.
- **SVC-23** — `analyticsRoutes` has no caller auth (`optionalStoreIdentification` only) → any shop's revenue/user analytics readable via `?shop=`. → add `requireAuth`.
- **SVC-16** — `ensureCustomer` is read-then-write with no upsert → concurrent first-bids race into `E11000`. → `findOneAndUpdate(..., {upsert:true})`.

### Admin / platform
- **ADMIN-02** — `onboardingComplete: ... || true` (`auction-admin/src/App.jsx:203`) is always truthy → onboarding never shows. → use `?? false`.
- **ADMIN-01** — `main.jsx` ships 5 `originalLog('🚀…')` calls that **bypass** the debug gate (fire in prod), with imports interleaved with executable statements. → remove logs, hoist imports.
- **PLAT-01** — `@react-router/serve` (used by `start`) is **not in `dependencies`**; `--omit=dev` prod install may not have it → boot crash. → add to `dependencies`, verify clean-container boot.

## P1 — HIGH (performance & reliability)

### Backend hot path & queries
- **BACKEND-03** — `getDailyStats` issues up to **365 sequential `find()`** for a 1y range. → one `$group` aggregation bucketed by day.
- **BACKEND-04** — `getAnalytics` loads **every auction** (full `bidHistory`/`productData`, no `.lean()`) into memory to count statuses. → aggregation + projection.
- **BACKEND-05** — `getAuctionStats` loads all auctions and recomputes status ~13×/auction; also includes soft-deleted. → compute once / aggregate, add `isDeleted:{$ne:true}`.
- **BACKEND-06** — Every bid does `io.to(room).emit` **and** a global `io.emit` of the full decorated auction to **all** connected sockets → fan-out storm + cross-tenant data leak. → drop the global emits; scope to rooms, slim payload.
- **SVC-06** — `POST .../auction-metafields` loops 10 metafields × (GET + PUT) = **~20 serial REST calls per bid**, blocking the bid response and burning the 40-call bucket. → single `metafieldsSet` GraphQL mutation.
- **BACKEND-12** — Auction mutations make the server **HTTP-call itself** (`fetch ${API_BASE_URL}/api/metafields/...`), re-entering Express/auth/store-lookup per bid. → extract to a service, call in-process.
- **BACKEND-11** — `refreshAllProductData` does `Auction.find({})` across **all tenants** with serial Shopify calls in a request handler → timeout + multi-tenancy violation. → scope by shop, batch, background job.
- **SVC-07** — `shopifyGraphQLService.executeGraphQL` has **no 429/THROTTLED retry** (unlike `shopifyService.executeWithRetry`). Duplication/winner-product creation fails hard under load. → add backoff.

### Front-end performance
- **CUST-01** — Widget polls every 5s even with socket connected (the guard checks `socket.connected` synchronously right after `io()`). → start polling only on `connect_error`/timeout, clear on `connect`.
- **CUST-04 / PERF-001** — Per-card `setInterval` countdowns: 50 auctions × up to 3 timers = 150 timers/sec → mobile jank. → one shared ticker context.
- **CUST-06** — After every bid the widget calls full `fetchVisibleAuctions()` (spinner + re-render all cards) although the bid response + socket event already carry new state. → update just the affected auction.
- **CUST-07** — `joinAuction` effect depends on `[auctions]` → re-emits `join-auction` for all auctions on every bid/poll. → track joined IDs in a ref.
- **ADMIN-03** — Dashboard socket effect keyed on `[i18n]` → tears down/re-registers listeners + `fetchStats()` storm on every locale/render change. → `[]`-keyed connection effect, read i18n via ref.
- **ADMIN-12** — `getShopFromURL()` (70-line multi-source resolver) runs **inside the request interceptor on every API call**. → memoize per session.
- **SVC-10** — Blast email `sendAllAtOnce` sends up to 10k emails in a tight loop, fire-and-forget with no `.catch` → SMTP ban + possible unhandledRejection crash. → throttle + `.catch`.
- **SVC-15** — `emailService` does sync `fs.existsSync`/`readFileSync` per template per email (blocking the event loop, repeated per recipient). → cache localized templates.
- **PLAT-02** — Root `build` compiles the unrelated standalone `auction-admin` via `buildWithSkip.js` → bloated deploys; an admin build failure aborts the Remix deploy. → decouple.

### Reliability
- **SVC-11** — `processAllEndedAuctions` returns `undefined` → cron always logs "nothing processed" and masks whether the safety net works. → return a count.
- **SVC-18** — Winner processing releases the lock on error but never advances state → cron **retries the same failing auction forever** (e.g. deleted product → 404). → add `processingAttempts` + `failed` terminal state.
- **SVC-14** — `emailService.sendEmail` swallows SMTP errors (returns `{success:false}`, ignored by callers) → failed winner/invoice emails are invisible. → surface/throw + retry.

- **PLAT-06/07** — `useAuthenticatedFetch` returns a new fn every render → unstable identity drives effect/`useCallback` loops (`AuctionDetails`, `AuctionForm`). → `useMemo(() => app ? authenticatedFetch(app) : fetch, [app])`.

## 🚀 Strategic upgrades — make Bidly best-in-class

Beyond fixing bugs, these are the structural moves that separate a "vibe-coded" app from an industry-leading one:

1. **Introduce a real job queue (BullMQ + Redis).** Today, auction-end, winner processing, emails, blast sends, and metafield syncs all run inline in the request path or in a fragile 5s `setInterval`. A queue gives you: atomic single-execution, retries with backoff, dead-letter handling, scheduled jobs, and horizontal scaling. This single change fixes the duplicate-email, infinite-reprocess, SMTP-flood, and blocking-email-send problems at the root.

2. **Move all storefront writes behind a single GraphQL batch + rate-limit-aware client.** Replace the ~20-REST-calls-per-bid pattern with one `metafieldsSet` mutation through a shared client that understands Shopify's cost/throttle extensions. Add a token-bucket so you never hit 429.

3. **One shared real-time layer, room-scoped, slim payloads.** Kill the global `io.emit`. Define a clear channel model (`auction:{id}`, `admin:{shop}`), send deltas not full documents, and add a Redis Socket.IO adapter so real-time survives multiple backend instances. This fixes the cross-tenant leak and lets you scale past one server.

4. **Aggregate, don't iterate.** Every analytics/stats endpoint should be a MongoDB aggregation pipeline with proper compound indexes (`{shopDomain, status, endTime}`, `{shopDomain, createdAt}`, `{status, completedAt}`). Add `.lean()` everywhere reads don't need documents. The dashboard goes from seconds to milliseconds.

5. **Consolidate the front-end story.** You currently ship: a React customer widget, a 2,818-line dead vanilla widget, a 370-line dead login script, and giant inline-HTML renderers in the backend. Pick **one** widget delivery path (the React widget), delete the rest, and minify/bundle the theme JS (it's ~8,700 unminified lines).
